# Data Models

This document describes both the PostgreSQL relational schema (managed by Prisma) and the Neo4j graph model (the food knowledge graph).

---

## PostgreSQL Schema

Defined in `backend/prisma/schema.prisma`. The provider is `postgresql` and the client is generated by `prisma-client-py`.

---

### User

The root entity. Identified by either phone number or email address.

```
Table: users
```

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | String (UUID) | Primary Key | Auto-generated UUID |
| `phone` | String | Unique, Nullable | Phone number used as identifier |
| `email` | String | Unique, Nullable | Email address used as identifier |
| `password_hash` | String | Not Null | bcrypt hash of the password |
| `language` | String | Default `bn` | Preferred language: `bn` or `en` |
| `created_at` | DateTime | Default now() | Account creation timestamp |

At least one of `phone` or `email` must be provided at registration. The application enforces this at the router level.

**Relations:**
- One-to-one with `Profile`
- One-to-many with `HealthLog`, `MealPlan`, `ChatMessage`, `MealTracking`, `MedicineReminder`

---

### Profile

Stores the user's health and dietary profile. Created after registration by the user through the onboarding flow.

```
Table: profiles
```

| Column | Type | Constraints | Description |
|---|---|---|---|
| `user_id` | String (UUID) | Primary Key, FK to users | One-to-one with User |
| `name_bn` | String | Nullable | Name in Bengali |
| `name_en` | String | Nullable | Name in English |
| `age` | Int | Nullable | Age in years |
| `gender` | String | Nullable | `male` or `female` |
| `weight_kg` | Float | Nullable | Current weight in kilograms |
| `height_cm` | Float | Nullable | Height in centimetres |
| `activity_level` | String | Nullable | `sedentary`, `light`, `moderate`, `active`, `very_active` |
| `goal` | String | Nullable | `weight_loss`, `weight_gain`, or `maintain` |
| `medical_conditions` | String | Nullable | JSON array: `["Diabetes", "Hypertension"]` |
| `preferred_foods` | String | Nullable | JSON array of preferred food names |
| `disliked_foods` | String | Nullable | JSON array of disliked food names |
| `updated_at` | DateTime | Auto-updated | Last modification timestamp |

List-type fields (`medical_conditions`, `preferred_foods`, `disliked_foods`) are stored as JSON strings in the database and deserialized by the `safe_list()` utility in `app/utils.py`.

### MealPlan

A generated meal plan for a specific date. Plans are generated daily and weekly by the AI service and stored for retrieval.

```
Table: meal_plans
```

| Column | Type | Constraints | Description |
|---|---|---|---|
| `plan_id` | String (UUID) | Primary Key | Auto-generated UUID |
| `user_id` | String (UUID) | FK to users | Owning user |
| `plan_date` | DateTime | Not Null | The date this plan is for |
| `plan_type` | String | Not Null | `daily` or `weekly` |
| `plan_data` | String | Not Null | Full plan as a JSON string |
| `calorie_target` | Int | Not Null | Computed daily calorie target |
| `ai_suggestion_cal` | Int | Nullable | Calorie target suggested by the AI |
| `user_choice_cal` | Int | Nullable | Calorie target chosen by the user if overridden |
| `language` | String | Not Null | Language of the plan content |
| `feedback` | Int | Nullable | User rating 1 to 5 |
| `completed_slots` | String | Nullable | JSON array of completed meal slots: `["breakfast", "lunch"]` |
| `created_at` | DateTime | Default now() | Record creation timestamp |

The `plan_data` JSON structure:
```json
{
  "meals": [
    {
      "slot": "breakfast",
      "slot_bn": "সকালের নাস্তা",
      "target_calories": 450,
      "items": [
        {
          "food_code": "rice_white",
          "name_en": "White Rice",
          "name_bn": "ভাত",
          "amount_g": 150,
          "calories": 195,
          "protein_g": 4.0,
          "carbs_g": 43.0,
          "fat_g": 0.5
        }
      ]
    }
  ]
}
```

### ChatMessage

A single message in the AI chat conversation, stored for history retrieval and context injection.

```
Table: chat_messages
```

| Column | Type | Constraints | Description |
|---|---|---|---|
| `message_id` | String (UUID) | Primary Key | Auto-generated UUID |
| `user_id` | String (UUID) | FK to users | Owning user |
| `role` | String | Not Null | `user` or `assistant` |
| `content` | String | Not Null | Full text of the message |
| `intent` | String | Nullable | Reserved for intent classification |
| `created_at` | DateTime | Default now() | Message timestamp |

The chat router saves both user and assistant turns to this table after each exchange. History is retrieved and re-injected into the LLM context on subsequent requests (last 10 turns).

### MealTracking

A single meal logged by the user, either through natural language input or photo upload.

```
Table: meal_tracking
```

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | String (UUID) | Primary Key | Auto-generated UUID |
| `user_id` | String (UUID) | FK to users | Owning user |
| `input_text` | String | Not Null | Original description with a verification status prefix |
| `parsed_items` | String | Not Null | JSON array of identified food items with nutritional data |
| `total_calories` | Int | Not Null | Sum of calories across all parsed items |
| `macros` | String | Not Null | JSON object: `{"protein_g": x, "carbs_g": x, "fat_g": x}` |
| `feedback` | String | Not Null | System-generated feedback string (success or not-found message) |
| `meal_slot` | String | Nullable | `breakfast`, `lunch`, `dinner`, or `snack` |
| `language` | String | Default `en` | Language of the input |
| `logged_at` | DateTime | Default now() | When the meal was logged |

The `parsed_items` JSON structure:
```json
[
  {
    "name": "ভাত",
    "amount_g": 150.0,
    "calories": 195.0,
    "protein_g": 4.1,
    "carbs_g": 43.0,
    "fat_g": 0.5
  }
]
```

All nutritional values in `parsed_items` are sourced exclusively from the Neo4j food database. LLM-estimated values are never stored.

When a meal is logged, the system also attempts to auto-complete the corresponding slot in today's `MealPlan` record.

## Neo4j Graph Model

The food knowledge graph is stored in Neo4j and queried via Bolt using Cypher. It encodes the National Dietary Guidelines (NDG 2025) as a structured graph of relationships between foods, nutrients, and medical conditions.

### Node Labels

**Food**

Represents a single food item from the Bangladeshi food composition table.

| Property | Type | Description |
|---|---|---|
| `code` | String | Unique food code (e.g. `rice_white`, `hilsa_fish`) |
| `name_en` | String | English name |
| `name_bn` | String | Bengali name |
| `name_original` | String | Original name from the data source |
| `energy_kcal` | Float | Calories per 100g |
| `protein_g` | Float | Protein per 100g in grams |
| `fat_g` | Float | Fat per 100g in grams |
| `carbohydrate_g` | Float | Carbohydrates per 100g in grams |
| `fiber_g` | Float | Dietary fiber per 100g in grams |
| `is_partial` | Boolean | True if the nutritional data is incomplete |

---

**FoodGroup**

Groups foods into categories matching NDG food group classifications.

| Property | Type | Description |
|---|---|---|
| `name_en` | String | Group name in English (e.g. `Cereals`, `Fish`, `Pulses`) |
| `name_bn` | String | Group name in Bengali |

---

**Nutrient**

A micronutrient tracked in the knowledge graph.

| Property | Type | Description |
|---|---|---|
| `name` | String | Nutrient name (e.g. `Iron`, `Calcium`, `Vitamin C`) |

---

**Condition**

A medical condition or dietary goal that influences food recommendations.

| Property | Type | Description |
|---|---|---|
| `name` | String | Condition name (e.g. `Diabetes`, `Hypertension`, `Anemia`, `Weight Loss`) |

---

**Disease**

A disease node linked to clinically required nutrients. Used by the RAG planner to prioritize micronutrient-rich foods.

| Property | Type | Description |
|---|---|---|
| `name` | String | Disease name |

---

**FoodAlias**

A synonym, brand name, or local variation that maps to a canonical Food node.

| Property | Type | Description |
|---|---|---|
| `name` | String | Alias or synonym |
| `multiplier` | Float | Conversion factor (e.g. for cooked vs raw weight) |

---

### Relationship Types

**`(Food)-[:BELONGS_TO]->(FoodGroup)`**

Classifies each food into exactly one food group.

---

**`(Food)-[:CONTAINS_NUTRIENT {amount_mg: Float}]->(Nutrient)`**

Declares the micronutrient content of a food. The `amount_mg` property stores the amount in milligrams per 100g of food.

---

**`(Condition)-[:AVOID_GROUP {reason: String}]->(FoodGroup)`**

Declares that users with this condition should avoid all foods in this group. The `reason` property contains a human-readable clinical explanation used in dietary rule display.

Example: `(Condition {name: "Diabetes"})-[:AVOID_GROUP {reason: "High glycemic index; raises blood sugar rapidly"}]->(FoodGroup {name_en: "Sweets"})`

---

**`(Condition)-[:PREFER_GROUP {reason: String}]->(FoodGroup)`**

Declares that users with this condition benefit from foods in this group. Foods in preferred groups receive a higher `preference_score` in safe food queries.

---

**`(Disease)-[:REQUIRES]->(Nutrient)`**

Declares which micronutrients are clinically important for managing a disease. Used by the chatbot context builder to highlight relevant micronutrient data for the user's conditions.

---

**`(FoodAlias)-[:MAPS_TO {multiplier: Float}]->(Food)`**

Resolves an alias to a canonical food. The `multiplier` adjusts nutritional values for conversion (e.g. a cooked alias with a different yield ratio than the raw food entry).

### Key Cypher Queries

**Search foods by name (Bengali or English):**
```cypher
MATCH (f:Food)-[:BELONGS_TO]->(fg:FoodGroup)
WHERE toLower(f.name_en) CONTAINS toLower($qt)
   OR toLower(f.name_bn) CONTAINS toLower($qt)
RETURN f.code, f.name_en, f.name_bn, f.energy_kcal, f.protein_g, fg.name_en
LIMIT 10
```

**Get safe foods for a user's conditions:**
```cypher
MATCH (f:Food)-[:BELONGS_TO]->(fg:FoodGroup)
WHERE f.is_partial = false
OPTIONAL MATCH (ca:Condition)-[:AVOID_GROUP]->(fg)
WHERE ca.name IN $conditions
WITH f, fg, ca
WHERE ca IS NULL
OPTIONAL MATCH (cp:Condition)-[:PREFER_GROUP]->(fg)
WHERE cp.name IN $conditions
RETURN f.code, f.name_en, f.name_bn, f.energy_kcal, fg.name_en,
       count(cp) AS preference_score
ORDER BY preference_score DESC, f.protein_g DESC
LIMIT 40
```

**Get full chatbot context for a food:**
```cypher
MATCH (f:Food)-[:BELONGS_TO]->(fg:FoodGroup)
WHERE f.code = $term OR toLower(f.name_en) CONTAINS toLower($term)
WITH f, fg LIMIT 1
OPTIONAL MATCH (c:Condition)-[r:AVOID_GROUP|PREFER_GROUP]->(fg)
WHERE c.name IN $conditions
WITH f, fg, collect({condition: c.name, rel_type: type(r), reason: r.reason}) AS rules
OPTIONAL MATCH (f)-[rc:CONTAINS_NUTRIENT]->(nut:Nutrient)
WITH f, fg, rules, collect({name: nut.name, amount: rc.amount_mg}) AS nutrients
OPTIONAL MATCH (d:Disease)-[:REQUIRES]->(req_nut:Nutrient)
WHERE d.name IN $conditions
RETURN f.name_bn, f.name_en, fg.name_en, f.energy_kcal, f.protein_g,
       f.fat_g, f.carbohydrate_g, f.fiber_g, rules, nutrients,
       collect(DISTINCT req_nut.name) AS required_nutrients
```

## Data Serialization Notes

Several columns in PostgreSQL store structured data as JSON strings because Prisma Client Python does not natively support array or JSON column types in all database providers. The `app/utils.py` module provides helpers for this:

- `to_json_string(value)` — serializes a Python list or dict to a JSON string for storage
- `from_json_string(value)` — deserializes a JSON string back to a Python object
- `safe_list(value)` — ensures a value is always returned as a Python list, handling both raw lists and JSON strings
- `safe_dict(value)` — ensures a value is always returned as a Python dict

Affected columns: `medical_conditions`, `preferred_foods`, `disliked_foods`, `symptoms`, `parsed_items`, `macros`, `times`, `completed_slots`, `plan_data`.
